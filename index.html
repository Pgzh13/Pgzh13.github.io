MySQL-B-MySQL锁机制

#MySQL高级

##锁

###数据库锁

--说明：其是计算机协调多个进程或线程并发访问某一资源的机制

--分类：
    a.从对数据操作的类型（读/写）分类：
        1-读锁(共享锁)：针对同一份数据，多个读操作可以同时进行而不会互相影响;
            <!-- 对MyISAM表的读操作（加读锁），不会阻塞其他进程对同一表的读请求，但会阻塞对同一表的写请求，只有当读锁释后，才会执行其他进程的写操作 -->

        2-写锁(排它锁)：当前写操作没有完成前，其会阻断其他写锁和读锁
            <!-- 会阻塞其他进程对同一表的读写操作，只有锁释放后，才会执行其他进程的读操作 -->


    b.对数据操作的粒度分：
        1-表锁(偏读)：
            （1）特点：
                --偏向MyISAM存储引擎，开销小，加锁快；
                --无死锁
                --锁定粒度大，发生锁冲突的概率最高，并发度最低
            （2）使用：
                    --手动增加表锁：
                        lock table 表名字 read(write),表名字2 read(write),其它; 
                    --查看表上加过的锁：
                        show open tables;
                    --释放表：
                        unlock tables;
        2-行锁(偏写)

--分析表锁定：
    a.说明：可以通过指定的状态变量来分析系统上的表锁定
    b.状态变量：
        1-Table_locks_imamediate：产生表级锁定的次数，表示可以立即获取锁的查询次数，每立即获取锁值+1
        2-Tables_locks_waited：出现表级锁定争用而发生等待的次数（不能立即获取锁的次数，每等待一次锁值+1），此值高则说明存在着较严重的表级锁争用的情况
            <!-- Myisam的读写锁调度是写优先，所以Myisam不适合做写为主表的引擎，因为在写锁后，其他线程不能做任何操作，大量的更新会使擦汗寻很难得到锁，从而造成永久阻塞 -->

MySQL-A-索引

#MySQL高级

##索引

###BTREE结构

--说明：多路平衡搜索树

--特性：以m叉的BTree树为例
    a.树中每个节点最多含m个孩子
    b.除根结点与叶子节点外，每个节点至少有[ceil(m/2)]个孩子
    c.若根节点不是叶子节点，则至少有两个孩子
    d.所有的叶子节点都在同一层
    e.每个非叶子节点由n个key与n+1个指针组成，其中[ceil(m/2)-1] <= n <=m-1

--B+TREE结构：
    a.与BTREE区别：
        1-n叉B+Tree最多含有n个key，而BTree最多含有n-1个key
        2-B+Tree的叶子节点保存所有key信息，依大小顺序排列
        3-所有非叶子节点都可以看作是key的索引部分

    b.优点：提高了区间访问的性能

###索引

--SQL解析：

--说明：索引（index）是帮助MySQL高效获取数据的数据结构（有序）

--优点：
    a.提高了数据的检索效率，降低了数据库的IO成本
    b.降低了数据排序的成本，降低了CPU的消耗

--缺点：
    a.索引列也会占用多余的空间
    b.降低了更新表的速度

--分类：
    a.单值索引：即一个索引只包含单个列，一个表可以有多个单列索引
    b.唯一索引：索引列的值必须唯一，但允许有空值
    c.复合索引：即一个索引包含多个列

--语法:
    a.创建索引：
        CREATE [UNIQUE | FULLTEXT |SPATIAL] INDEX index_name
            [USING index_type]
        ON tbl_name(index_col_name,.......)
        index_col_name : column_name[(length)][ASC | DESC]

    b.修改索引：
        ALTER tbl_name ADD [UNIQUE] INDEX [index_name] ON column_name(length)]

    b.查看索引：
        show index from table_name【\G，可以用于查看不同的索引】

    c.删除索引：
        drop index index_name on table_name

--适用情况：
    a.主键自动创建唯一索引
    b.频繁作为查询条件的字段应该创建索引
    c.查询中与其他关联的字段，外键关系建立索引
    d.频繁更新的字段不适合创建索引
    e.where条件里用不到的字段不创建索引
    f.单键/组合索引的选择问题（在高并发下倾向创建组合索引）
    g.查询中排序的字段，排序字段若通过索引去访问将大大提高排序速度
    h.查询中统计或分组的字段

###性能分析

--EXPLAIN关键字：
    a.说明：该关键字可以模拟执行SQL语句，从而指导MySQL是如何处理你的SQL语句，从而分析你的查询语句或是表结构的性能瓶颈

    b.使用：explain+SQL语句

    c.数据：
        1-id：
            说明：selet查询的序列号，包含一组数字，标识查询中执行select子句或操作表的顺序

            情况：
                （1）id相同，执行顺序由上到下
                （2）id不同，如果是子查询，id的序号会递增，id值越大优先级越高，越先被执行
                （3）id相同不同，同时存在

        2-selet_type：
            说明：查询的类型，主要用于区别普通查询、联合查询、子查询等的复杂查询

            分类：
                （1）SIMPLE：简单的select查询，查询中不包含子查询或者UNION
                （2）PRIMARY：查询中若包含任何复杂的子部分，最外层查询则被标记为该关键字
                （3）SUBQUERYL：在select或where列表中包含了子查询
                （4）DERVED：在from列表中包含的子查询被标记为DERIVED（衍生），MySQL会递归执行这些子查询，把结果放在临时表中
                （5）UNION若第二个select出现在UNION之后，则会被标为为UNION，若UNION包含在from子句的子查询中，则外层select将被标记为DERIVED
                （6）UNION RESULT：从UNION表中获取结果的select

        3-type：
            说明：访问类型排序
                <!-- 一般来说，得保证至少达到range级别，最好能达到ref -->
            （1）级别顺序大小：system > const > eq_ref > ref > range > index > ALL
            （2）分类：
                --system：表只有一行记录（等于系统表），这是const类型的特列
                --const：标识通过索引一次就找到了。const用于比较primary key或者unique索引
                --eq_ref：唯一性索引扫描，对于每个索引键，表中只有一条记录与之匹配。常见于主键或唯一索引扫描
                --ref：非唯一性索引扫描，返回匹配某个单独值的所有行，本质上也是一种索引访问，他返回所有匹配某个单独值的行，然而，他可能会找多个符合条件的行，所以他应该属于查找和扫描的混合体
                --range：只检索给定范围的行，使用一个索引来选择行。key列显示使用了哪个索引一般就是在你的where语句中出现了between、>、<、in等的查询
                --index：Full index Scan，index与ALL的区别为index类只遍历索引树（虽然其与ALL都是全表扫描，但是其是从索引中读取，而all是从硬盘中读取）
                --all：将遍历全表以找到匹配的行

        4-possible_keys：
            说明：显示可能应用在这张表中的索引，一个或多个

            情况：查询涉及到的字段上若存在索引，则该索引将被列出，但不一定被查询实际使用

        5-key：
            说明：实际使用的索引
            情况：
                --如果为null，则没有使用索引；
                --查询中若使用了覆盖索引，则该索引仅出现在key列表中

        6-key_len：
            说明：表示索引中使用的字节数，可通过该列计算查询中使用的索引长度

            情况：
                （1）在不损失精确性的情况下，长度越短越好
                （2）key_len显示的值为索引字段的最大可能长度，并非实际使用长度，即key_len是根据表定义计算而得，不是通过表内检索出的

        7-ref：显示索引的哪一列被使用

        8-rows：根据表统计信息及索引选用情况，大致估算出找到所需的记录所需要读取的行数

        9-Extra：包含不适合在其他列中显示但十分重要的额外信息
            分类：
                （1）Using filesort：说明mysql会队数据使用一个外部的索引排序，而不是按照表内的索引顺序进行读取。MySQL中无法利用索引完成的排序操作称为“文件排序”
                （2）Using temporary：实用了临时表保存中间结果，MySQL在对查询解惑排序时使用临时表。常见于排序order by和分组查询group by
                （3）Using index：表示相应的select操作中使用了覆盖索引，避免访问了表的数据行，效率不错。如果同时出现using where，表明索引被用来执行索引键值的查找；如果没有同时出现using where，表明索引用来读取数据而非执行查找动作
                    <!-- 覆盖索引：可以解释为select的数据列只用从索引中就能够获取得，不必读取数据行，MySQL可以利用索引返回select列表中的字段，而不必根据索引再次读取数据文件，等言之查询列要被所建的索引覆盖  -->
                （4）Using where：表明使用了where过滤
                （5）Using join buffer：使用了连接缓存
                （6）impossible where：where子句的值总是false，不能用来获取任何元组
                （7）select tables optimized away：在没有group by句子的情况下，基于索引优化MIN/MAX操作或者对于MyISAM存储引擎优化count(*)操作，不必等到执行阶段再进行计算
                （8）distinct：优化distinct操作，在找到第一匹配的元组后即停止找同样值的动作

###查询优化

--索引分析：
    a.join语句的优化：
        1-尽可能减少join语句中的NestedLoop的循环总次数；永远用小结果集驱动大的结果集
        2-优先优化NestedLoop的内层循环
        3-保证join语句中被驱动表上join条件字段已经被索引

--索引失效：
    a.尽量使用全值匹配
    b.最佳左前缀法则
    c.不在索引列上做任何操作（计算、函数、(自动或手动)类型转换），会导致索引失效而转向全表扫描
    d.存储引擎不能使用索引中范围条件右边的列
    --尽量使用覆盖索引（只访问索引的查询(索引和查询列一致)），减少select*
    e.mysql在使用不等于的时候无法使用索引会导致全表扫描
    f.id null,is not null也无法使用索引
    g.like以通配符开头('%abc...')mysql索引是小会变成全表扫描的操作，可以通过覆盖索引解决该问题
    h.字符串不加单引号索引失效
    i.减少使用or的频率，用其连接时会索引失效

--查询截取分析： 
    a.查询优化：
        1-优先原则：小表驱动大表，即小的数据集驱动大的数据集

    <!-- 
        exists关键字：
            1-语法：select .. from table where exists(subquery)
            2-作用：可以将该语法看作，将主查询的数据，放到子查询中做条件验证，根据验证结果来决定著查询的数据结果是否得以保留
            3-注意：结果返回true或false，因此子查询的查询列表可以是一个常量，即实际执行会忽略select清单，因此没有区别
    -->

    b.order by子句：尽量使用index方式排序，避免使用FilieSort方式排序
        1-排序方式：
            （1）文件排序
            （2）扫描有序索引排序

        2-排序使用索引情况：
            （1）order by使用索引最左前缀
            （2）where使用索引的最左前缀定义为常量，则order by能使用索引

        3-排序不能使用索引的情况：
            （1）排序不一致
            （2）没有遵循最左前缀原则
            （3）对于排序来说，多个相等条件也是范围查询
            （4）对索引之外的字段进行排序

        4-提高order by的速度：
            （1）尽量使用覆盖索引，避免使用select*
                --当查询的字段大小总和小于max_length_for_sortData而且排序字段不是TEXT|BLOB类型时，会用改进后的算法(单路排序)，否则会使用回多路排序
                --两种算法的数据都有可能超出sort_buffer的容量，超出之后，会创建tmp文件进行合并排序，导致多次I/O，但是用单路排序算法的风险会更加大，可以通过提高sort_buffer_size来解决
            （2）尝试提高sort_buffer_size
            （3）尝试提高max_length_for_sortData

    c.group by子句：其优化方式与order by大致相同；
        1-注意：
            （1）group by实质时先进行排序后分组，准寻索引建的最佳左前缀
            （2）当无法使用索引列，同时通过增大max_length_for_sortData参数和sort_buffer_size参数的设置
            （3）where高于having，能写在where限定的条件就不要在having当中去限定

MySQL-存储过程和函数

#数据库

##系统存储

###变量

--系统变量：
    a.说明：变量由系统提供，不是用户定义，属于服务器层面

    b.使用语法：
        1-查看所有的系统变量:
            show global |【session会话】variables;
        2-查看满足条件的部分系统变量
            show global |【session】variables like '%char%';
        3-查看指定的某个系统变量的值
            select @@global | 【session】.系统变量名；
        4-设置系统变量的值
            方式一：
                set global | 【session】系统变量名 = 值;
            方式二：
                set @@global | 【session】 .系统变量名 = 值;

    c.作用域：
        1-全局变量：服务器每次启动将为所有的全局变量赋初始值，针对所有的会话（连接）有效，但不能跨重启
        2-会话变量：仅仅针对当前会话（连接）有效

--自定义变量：
    a.说明：变量是用户自定义的，不是由系统的

    b.分类：
        1-用户变量：
            （1）作用域：针对当前会话（连接）有效，同于会话变量的作用域，应用在任何地方
            （2）使用：
                --声明或初始化：
                    set @用户变量名 =值 / :=值
                    select @用户变量名 =值 / :=值

                --赋值：
                    方式一：
                        使用set或select
                    方式二：
                        select 字段 into 变量名 from 表;

                --查看：
                    select @用户变量名;

        2-局部变量：
            （1）作用域：仅仅在定义它的【begin end】中有效，应用在【begin end】中的第一句话
            （2）使用：
                --声明：
                    declare 变量名 类型 【default 初始值】;

                --赋值：与用户变量的方式一致，但不需要@符号

                --查看
                    select 局部变量名

###存储过程

--说明：一组预先编译好的SQL语句的集合，理解成批处理语句

--使用：
    a.创建：
        create procedure 存储过程名(参数列表)
            begin
                存储过程体(
                        --一组合法的SQL语句，如果只有一条SQL语句则【begin end】可以省略
                        --其每条SQL语句阿结尾要求必须加分号
                        --存储过程的结尾可以使用delimiter重新设置，语法：delimiter 结束标记
                    )
            end
        <!-- 
            参数列表：
                --参数模式：
                    a.in：该参数可以作为输入，也就是该参数需要调入方传入值
                    b.out：该参数可以作为食醋胡，也就是该参数可以作为返回值
                    c.inout：该参数既可以作为输入又可以作为输出，也就是该参数既需要传入值，也可以返回值
                --参数名
                ---参数类型

         -->

    b.调用：
        call 存储过程名(实参列表);

    c.删除：
        drop procedure 存储过程名;

    d.查看：    
        show create procedure 存储过程名

    <!--

    a.if函数

    b.case结构

    c.loop结构：
        1-语法：

        1-语法：

            end repeat 【标签】;

#数据库

##事务控制语言

###事务

--说明：事务由单独单元的一个或多个SQL语句组成，在这个单元中，每个MySQL语句时相互依赖的

--事务的ACID属性：
    a.原子性（Atomicity）：指事务是一个不可分割的工作单位
    b.一致性（Consistency）：事务必须使数据库从一个状态变换成另外一个一致性状态
    c.隔离性（Isolation）：指一个事务的执行不能被其他事务干扰，即一个事务内部的操作及使用的数据队并发的其他事务时隔离的
    d.持久性（Durability）：指一个事务一旦被提交，他对数据库中数据的改变是永久性的，接下来的其他操作和数据库故障不应该对其有任何影响

--分类：
    a.隐式事务：事务没有明显的开启和结束标志
    b.显示事务：事务具有明显的开启和结束的标记
        1-前提：必须先设置自动提交功能为禁用【set autocommit = 0】
--事务的使用：
    a.语法：
        【
            set autocommit = 0;  //设置自动提交功能为禁用
            start transaction;  //可选的，开启事务
                <!-- 编写sql语句 -->
            commit;     //提交事务
            rollback;   //回滚事务，可以通过savepoint设置一个保存点，通过该关键字回滚到指定的保存点
        】

    b.并发问题：    
        1-脏读：对于两个事务t1、t2，t1读取了已经被t2更新但还没有被提交的字段，之后，若t2回滚，t1读取的内容就是临时却无效的
        2-不可重复读：对于两个事务t1、t2，t1读取了一个字段，然后t2更新了该字段，之后，t1再次读取同一个字段，值就不同了
        3-幻读：对于两个事务t1，t2，t1从一个表中读取了一个字段，然后t2在该表中插入了一些新的行，之后，如果t1再次读取同一个表，就会多出几行

    c.事务的隔离级别：
        1-Oracle只支持2种事务隔离级别：READ COMMITED（默认），SERIALIZABLE
        2-MySQL支持4种事务隔离级别，默认为REPEATABLE READ

    d.注意：事务当中执行删除操作时，delete是支持回滚的，而truncate是不支持回滚的

    d.事务常用命令：
        1-查看当前的隔离级别：
            SELECT @@tx_isolation
        2-设置当前mysql连接的隔离级别：
            set transaction isolation level read committed
        3-设置数据库系统的全局的隔离级别：
            set global transaction isolation level read committed

###视图

--说明：虚拟表，和普通表一样使用，mysql15.1版本出现的新特性，是通过表动态生成的数据，只保存了sql逻辑，不保存查询结果

--使用：
    a.创建：
        create view 视图名 as 查询语句   

    b.修改：
        1-方式一：
            create or replace view 视图名 as 查询语句
        2-方式二：
            alter view 视图名 as 查询语句

    c.删除：
        drop view 视图名,视图名......;

    d.查看：
        1-show create view 视图名
        2-desc 视图名

    e.更新：
        1-使用：其增删改查与原表一致，而且会对原表进行更新
        2-限制：以下类型的视图是不能更新的
            （1）包含以下关键字的sql语句：分组函数、distinct、group by、having、union或者union all
            （2）常量视图
            （3）Select 中包含子查询
            （4）join(只可以进行更新，不能进行其他更新操作)
            （5）from一个不能更新的视图
            （6）where子句的子查询引用了from子句中的表
